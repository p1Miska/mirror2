# Client Mirror — Forge 1.12.2 mod

Read-only/интерактивный клиент-зеркало реального сервера через WebSocket-мост
(`server.js`, Mineflayer). Мод запускается в обычном локальном (плоском/void)
мире и отображает происходящее на реальном сервере через `ws://localhost:8081`.

## Сборка

### Вариант A — GitHub Actions (без локального JDK 8/тулчейна)

1. Создайте пустой репозиторий на GitHub.
2. Залейте туда **содержимое** этой папки (`ClientMirrorMod/*`) как корень репозитория —
   то есть `build.gradle` и `.github/` должны лежать прямо в корне репо, а не во
   вложенной папке:
   ```bash
   cd ClientMirrorMod
   git init
   git add .
   git commit -m "init"
   git branch -M main
   git remote add origin https://github.com/<you>/<repo>.git
   git push -u origin main
   ```
3. Во вкладке **Actions** репозитория запустится workflow `Build Forge mod`
   автоматически (или нажмите "Run workflow" вручную).
4. После завершения (обычно 5–10 минут на первый прогон, дальше кэш Gradle
   ускоряет) откройте завершённый run → **Artifacts** → скачайте `clientmirror-jar`.
   Внутри — готовый `.jar`, кладёте его в `mods/` вашего Forge 1.12.2 клиента.

Workflow использует `setupCiWorkspace` (лёгкая CI-версия деобфускации Minecraft,
без полного декомпилятора) — это специально сделано для таких раннеров, полный
`setupDecompWorkspace` (нужен только для удобной разработки в IDE) в CI не гоняем.

### Вариант B — локально

Нужен JDK 8 (ForgeGradle 2.3 не работает на JDK 9+).

```bash
cd ClientMirrorMod
./gradlew setupDecompWorkspace   # один раз, долго
./gradlew build
# результат: build/libs/clientmirror-1.0.0.jar
```

Jar собирается с шейднутой библиотекой `Java-WebSocket` внутри (см. `jar { from ... }`
в build.gradle), отдельно её ставить не нужно.

## Запуск

1. Запустите `server.js` (мост должен слушать `ws://localhost:8081`).
2. Установите собранный jar в `mods/` обычного Forge 1.12.2 клиента (или запускайте
   через `./gradlew runClient` для разработки).
3. Создайте/откройте ЛОКАЛЬНЫЙ одиночный мир (рекомендуется плоский/void пресет —
   реальный ландшафт всё равно будет затёрт данными с сервера). Мод сам подключится
   к WS при входе в мир и отключится при выходе.
4. Адрес моста и пороги коррекции позиции настраиваются в `config/clientmirror.cfg`.

## Статус по этапам (см. постановку задачи)

- [x] Этап 0 — каркас: WS-клиент с реконнектом, очередь сообщений, диспетчер, конфиг
- [x] Этап 1 (MVP) — `blocks`/`blockChange` → `world.setBlockState`, маппинг блоков
      1.21→1.12.2 с fallback (`world/BlockMapper.java`), подстилка барьером под
      непереданной сервером областью
- [x] Этап 2 — другие игроки как `EntityOtherPlayerMP` с постоянным ником,
      позиции обновляются по `state` (`entity/RemotePlayerManager.java`)
- [x] Этап 3 — локальная физика как основная, отправка `input`/`look`,
      мягкая коррекция при среднем расхождении, жёсткий телепорт/ресинк при
      большом расхождении или по `status: spawned` (обрабатывает переход между
      серверами через Bungee/Velocity без перезапуска клиента)
- [x] Этап 4а — перехват `dig`/`interact`, отмена локального изменения,
      применение блока только по `blockChange` от сервера; чат в обе стороны
- [x] Этап 4б — предметы (`item/ItemMapper.java`, аналог BlockMapper: незерит
      → алмаз, эльтра идёт напрямую т.к. уже есть в 1.12.2). Хотбар/броня/полный
      инвентарь зеркалятся в НАСТОЯЩИЙ инвентарь локального игрока
      (`item/InventoryMirror.java`) — ванильный HUD/GuiInventory рисуют это сами.
      Чужие контейнеры открываются в **настоящих ванильных GUI**, подставляя
      `RemoteInventory` вместо тайл-энтити:
        - `chest`/`shulker_box`/`barrel` → `RemoteGuiChest` (обёртка над `GuiChest`,
          сам считает число рядов из размера инвентаря — подходит и для
          одиночного, и для двойного сундука)
        - `furnace`/`smoker`/`blast_furnace` → `RemoteGuiFurnace`
        - `dispenser`/`dropper` → `RemoteGuiDispenser`
        - `brewing_stand` → `RemoteGuiBrewingStand`
        - всё остальное (`hopper` — `GuiHopper` не резолвится в текущем
          маппинге, см. чейнджлог ниже; `crafting_table`, `anvil`,
          `enchanting_table`, и блоки, которых в 1.12.2 вообще нет —
          `grindstone`/`stonecutter`/`loom`/`smithing_table`/`lectern`,
          добавлены в 1.14+) → универсальный fallback `GuiRemoteContainer`
          (нейтральная сетка слотов)
      Клики во всех случаях перехватываются (`mouseClicked` переопределён) и
      уходят как `windowClick` в WS; локально ничего не двигается — обновление
      только по следующему `window` от сервера.
      **Известное ограничение:** у печки/пивоварни синхронизированы только
      слоты — полоска прогресса варки/огонёк топлива статичны, потому что
      `RemoteInventory.getField(...)` всегда возвращает 0 (сервер пока не шлёт
      `burnTime`/`cookTime`/т.п. в `window`). Если нужно — доработать
      `serializeWindow` в `server.js`, чтобы прокидывать поля прогресса
      печи/пивоварни из mineflayer, и добавить их разбор в `RemoteInventory`.
- [x] Атака игроков — `AttackEntityEvent` перехватывается в
      `event/BlockInteractionHandler.java`, ЛКМ по чужому (замирроренному)
      игроку шлёт `attack` в WS. Атака мобов сервером сейчас не поддерживается
      (`server.js` резолвит цель только по `bot.players[username]`, т.е.
      только реальных игроков).
- [x] Медь — приоритет отдан ФОРМЕ, а не цвету: `cut_copper_stairs`/
      `copper_slab`/`copper_door` идут в деревянные аналоги (форма верная —
      ступенька/полублок/дверь остаются проходимыми; в 1.12.2 металлической
      плиты/лестницы физически не существует, выбора между "форма верная,
      цвет нет" и "цвет ближе, форма сломана" тут нет — только первый вариант
      играбелен). В `iron_block` уходят только "цельные" медные блоки без
      формового суффикса (`copper_block`, `cut_copper`, `copper_bulb` и т.п.),
      которые и в оригинале целый куб.
- [x] **Сборка (CI) — полная переработка тулчейна.** Первые попытки чинить
      версию Forge (`2860` → `2859`) не помогли, потому что дело было не в
      конкретной версии: начиная примерно с билда `2851–2860` Forge тихо
      перевёл 1.12.2 на новую систему сборки (ForgeGradle 5+), а официальную
      поддержку версии к этому моменту уже свернули — из-за чего часть
      мелочей осталась недочинена (`forge-userdev.jar` в старом
      classifier-формате больше не публикуется вообще ни для одной сборки).
      Это известная, задокументированная проблема самого MDK для 1.12.2 (есть
      отдельная community wiki-страница "Forge 1.12 MDK Fix"), а не косяк
      конкретно этого проекта. Текущий рабочий тулчейн — **ForgeGradle 5.1 +
      FancyGradle** (community-плагин, патчащий недочинённые Forge мелочи для
      1.12.2), под Gradle 7.5. Именно так собираются несколько реальных живых
      1.12.2-модов на GitHub (WizardryFates, mo-spells и т.п.) на момент
      написания. `build.gradle` и `.github/workflows/build.yml` переписаны
      под эту схему. Уточнение по ходу: URL репозитория FancyGradle
      `maven.gofancy.wtf/releases` оказался мёртвым/неверным — реальный,
      подтверждённый тремя живыми репозиториями адрес —
      `https://gitlab.com/api/v4/projects/26758973/packages/maven`
      (GitLab Packages), он и прописан в `build.gradle` сейчас.
- [x] Шейдинг Java-WebSocket в jar — в Gradle 7+ конфигурацию `implementation`
      нельзя резолвить напрямую (`canBeResolved=false`, так и задумано новой
      моделью конфигураций). Заведена отдельная `shade`-конфигурация
      специально для сборки jar, подключённая к `implementation` через
      `extendsFrom` — компиляция работает как обычно, а `jar { from {...} }`
      резолвит уже её.
- [x] **Реальные ошибки компиляции Java (первый раз дошли до этого этапа) —
      все исправлены.** Конкретный маппинг `snapshot_20171003` называет часть
      вещей иначе, чем можно было предположить по памяти/по более новым
      версиям:
        - `Items.WOODEN_DOOR` не существует — в 1.12.2 нет общего "деревянного"
          айтема двери, только породные варианты; исправлено на `Items.OAK_DOOR`
          (как и было изначально в `BlockMapper` для блока — тут просто забыл
          привести к единому в `ItemMapper`).
        - `Blocks.SIGN` не существует — блок стоящей таблички называется
          `Blocks.STANDING_SIGN` (в отличие от айтема, который `Items.SIGN`).
        - `World.addEntityToWorld(...)` в этом маппинге называется
          `World.spawnEntity(...)`.
        - `Vec3d.add(double, double, double)` не существует — принимает только
          `Vec3d`; исправлено на `eyes.add(new Vec3d(...))`.
        - `net.minecraft.client.gui.inventory.GuiHopper` не резолвится в этом
          маппинге вообще (под каким именем он доступен — не проверял, чтобы
          не гадать ещё раз) — `RemoteGuiHopper` полностью убран, воронка
          (`hopper`) теперь идёт на универсальный fallback `GuiRemoteContainer`
          так же, как crafting_table/anvil/enchanting_table.

## Известные ограничения (осознанные, не баги)

- Сервер отдаёт только **видимые** блоки (`isSolid` + открытая грань) в радиусе
  вокруг игрока и не чаще раза в ~800мс — рендер мира вокруг игрока неизбежно
  "рваный" по сравнению с настоящим сервером; это ограничение протокола, не мода.
- Маппинг блоков 1.21→1.12.2 приблизительный (см. `BlockMapper`), многие блоки
  сведены к ближайшему виду или fallback `stone`.
- Полная синхронизация инвентаря/контейнеров (этап 4б) не реализована.
